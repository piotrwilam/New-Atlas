# A Cross-Language Circuit Atlas for Code Models

**Draft v1 — 9 April 2026**

Target: EMNLP 2026 (ARR May 25 submission)

---

## Abstract (~250 words)

How do code language models internally represent programming languages, and does that representation depend on the language, the model, or both? We introduce a methodology for extracting, decomposing, and comparing concept-specific neural circuits across languages and models, producing a standardised atlas of circuit structure. Applying it to Python (106 concepts) and Rust (75 concepts) on Qwen2.5-Coder-7B and DeepSeek-Coder-V1-6.7B, we discover a dissociation between two independent dimensions of circuit structure. The "what" dimension — which concepts earn dedicated circuitry and how much — is task-determined: concept-specificity rankings correlate across independently trained models (Spearman ρ = 0.64 for Python, ρ = 0.72 for Rust), and 6 of 7 semantically equivalent concept pairs share >10% of concept-only neurons across Python and Rust within the same model. The "where" dimension — the depth at which concept-specific processing occurs — is training-determined: DeepSeek peaks at layers 4–8, Qwen at layers 14–19, with layer-profile correlation ρ = −0.14. Three specific findings emerge: (1) the atomicity super-cluster discovered in an 8-layer sparse transformer replicates in a dense 28-layer 7B model, confirming it is task-determined; (2) Rust concepts receive 3× more concept-specific circuitry than Python equivalents (mean concept fraction 0.215 vs 0.074), with the largest boosts for constructs that are syntactically weak in Python; (3) Rust's internal organisation in Qwen clusters into four semantically coherent groups — type-system traits, memory/ownership, data definition, and control-flow — mirroring the language's own conceptual taxonomy.

## 1. Introduction (~1.5 pages)

**Opening problem.** Code LLMs process dozens of programming languages, but we lack tools for asking comparative questions about their internal representations. When a model processes a `for` loop in Python and a `for` loop in Rust, does it use the same neurons? When two independently trained models both handle Python's `import`, do they allocate the same depth of circuitry? These questions require a methodology that produces comparable measurements across languages and models.

**Prior work: CSP-Atlas.** We previously showed that a sparse 8-layer Python transformer develops dedicated neural circuits for every syntactic construct and builtin object tested — a 100% survival rate across 106 concepts, 63,800 prompts, and 9 parameter settings. A cross-section experiment decomposed each circuit into concept-only neurons (responding to structural meaning but not the bare keyword token), shared neurons (responding to both), and token-only neurons. The key structural finding was an "atomicity super-cluster": Import, Break, Pass, Continue, Assert share circuitry not because they are semantically related, but because they are computationally atomic — single-statement constructs requiring no nested body.

**Gap.** CSP-Atlas was limited to one language (Python), one model (a sparse 8-layer transformer), and neuron-level analysis. Three questions remained open: (a) Do universal circuits survive in dense production-scale models? (b) Is the concept-vs-token decomposition a property of the model or the language? (c) Do different languages produce qualitatively different internal organisations?

**This paper.** We present the Atlas 2×2 methodology: a standardised pipeline for extracting, decomposing, and comparing neural circuits across languages and models. We apply it to Python (106 concepts) × Rust (75 concepts) × Qwen2.5-Coder-7B × DeepSeek-Coder-V1-6.7B. The methodology produces a directly comparable decomposition — concept-only fraction, meta-circuit structure, flow typology, cross-language sharing — from any (language, model) pair.

The cross-language, cross-model comparison reveals a central dissociation:

- **What** the model dedicates circuitry to is task-determined (conserved across models).
- **Where** the model processes it is training-determined (diverges across models).

This dissociation reframes the "are circuits universal?" question: circuit structure has at least two independent axes, and they have different answers.

Three specific findings follow from this dissociation:

**F1: Architecture-invariant organisation.** The atomicity super-cluster (Assert, Break, Continue, Import, ImportFrom, Pass) — discovered in a sparse 8-layer model — replicates exactly in a dense 28-layer 7B model (Qwen) and its functional analogue (Super, Use) appears in Rust. This is direct evidence that the grouping principle is task-determined.

**F2: Language design determines representation depth.** Rust concepts receive 3× more concept-specific circuitry than their Python equivalents (mean concept fraction 0.215 vs 0.074). The effect is largest for constructs that are syntactically weak in Python: Return (7.2×), For (5.1×), If (4.4×). Rust's stricter syntax forces the model to build more dedicated representations.

**F3: Rust's internal organisation mirrors its conceptual taxonomy.** Hierarchical clustering of Rust concept-only neuron sets at layer 14 reveals four groups: type-system traits (Enum, Send, Option, Copy, Eq, Drop, Debug), memory/ownership (Pin, Mutex, Arc, Box, Vec, Impl, Move, Future), data definition (Struct, Let, String, Trait, Default, Display), and control-flow/module keywords (Fn, Pub, Use, Crate, While, Break, Loop, If, Match). The model has discovered Rust's own semantic categories.

**Causal validation.** Ablation confirms the decomposition is not epiphenomenal: zeroing concept-only neurons produces statistically significant drops in target keyword probability (31 Bonferroni-significant concept×layer pairs across 6 of 7 testable Python concepts on Qwen).

**Concurrent work.** Krishnan et al. (2026) apply variance partitioning and sparse autoencoders to the same sparse CSP transformer, studying Python AST representations only. Their approach is complementary: different methods (variance partitioning, SAE feature discovery) on a different model class (sparse) for a single language. Our contribution is broader (cross-language, cross-model, dense architectures) and asks different questions (concept-vs-token separation as a function of language design).

## 3. Methodology (~1.5 pages)

### 3.1 Models

| Model | Params | MLP type | Hidden dim | Layers | Training tokens |
|-------|--------|----------|------------|--------|-----------------|
| Qwen2.5-Coder-7B | 7B | SwiGLU | 3,584 | 28 | 5.5T (92 langs) |
| DeepSeek-Coder-V1-6.7B | 6.7B | Dense | 4,096 | 28 | 2T (87 langs) |

Both dense (not MoE). Hooks on MLP output at every layer, last-token position.

**SwiGLU note.** For SwiGLU architectures, activation outputs are almost never exactly zero, so firing rates are effectively binary (0 or 1.0). The consistency threshold C becomes a no-op. The effective parameter sweep is over ε only: {0.001, 0.1, 0.5}. At ε = 0.001, circuits are trivially full. At ε = 0.1, concept-only fractions are near-zero. Meaningful concept-only signal emerges at ε = 0.5, indicating concept-only neurons are high-amplitude units. All results reported at ε = 0.5 unless stated otherwise.

### 3.2 Concept Spaces

| Language | Concepts | Testable | Types |
|----------|----------|----------|-------|
| Python (Qwen) | 106 | 58 | 43 AST nodes, 63 builtins |
| Python (DeepSeek) | 58 | 58 | Same 58 testable concepts |
| Rust (Qwen) | 75 | 57 | Keywords, types, traits, containers, ownership |

"Testable" = concept has a corresponding token that can appear in non-structural contexts (strings, comments, variable names), enabling the checker prompt contrast.

### 3.3 Contrastive Prompt Design

Two prompt types per concept:

- **Type 1 (concept prompts):** Executable code where the concept appears in its structural role. 50 prompts per (concept, context) pair with variance injected along lexical, structural, and padding dimensions.
- **Type 2 (checker prompts):** The keyword token appears but NOT in its structural role — in strings, comments, or variable names. Same variance injection.

Quality filtering by sequence loss.

### 3.4 Extraction Pipeline

Forward pass → hook MLP output (last token) → epsilon threshold (neuron active if |activation| > ε) → consistency filter (active in ≥ C% of prompts) → marginalisation (intersection across complementary dimension → universal circuit).

### 3.5 Three-Way Decomposition

Universal circuit mask (A) vs. checker mask (B):
- **Concept-only** (A \ B): neurons responding to structural meaning, not the bare token.
- **Shared** (A ∩ B): neurons responding to both.
- **Token-only** (B \ A): neurons responding only to the token.

Concept fraction = |A \ B| / |A|. Computed per concept, per layer.

### 3.6 Meta-Circuit Detection

Pairwise Jaccard similarity of concept-only neuron sets → hierarchical clustering (Ward linkage) → comparison with known grammatical categories.

### 3.7 Cross-Language Sharing

For each equivalence class (e.g., Python For ↔ Rust for), compute sharing fraction = |intersection of concept-only sets| / |smaller pool| at each layer. Permutation null model for significance.

### 3.8 Causal Validation

Per-layer zero-ablation of concept-only neurons vs. same-count random subsets of shared neurons vs. random neurons outside both sets. Metric: Δlog P(target keyword). Bonferroni correction across all concept × layer tests.

## 4. Finding 1: The What/Where Dissociation (~1.25 pages)

### 4.1 Calibration

Before presenting cross-model results, we verify that the decomposition recovers known positives and known negatives.

**Known positive.** The atomicity super-cluster — {Assert, Break, Continue, Import, ImportFrom, Pass} — discovered in the 8-layer sparse CSP transformer replicates exactly in Qwen as the only Python concepts classified two_phase in the flow typology (§6.1 presents the full analysis). The same grouping principle appears in Rust: Super and Use, both module-system keywords, are the only Rust two_phase concepts.

**Known negatives.** Python builtins (len, range, sorted, ...) have near-zero concept fractions in both models (Qwen: group mean 0.024; DeepSeek: 0.068), confirming that token-driven concepts correctly produce no concept-only signal. This holds for all 34 builtins tested.

**Predicted null.** Rust ownership operators (`&`, `*`, `'`) are tokens where the concept and the token are inseparable — there is no way to write the token without invoking the concept. The decomposition correctly returns near-zero concept fractions for these. The instrument measures what it claims.

### 4.2 The "What" Is Conserved Across Models

With calibration established, we compare concept-fraction rankings across two independently trained models. Neither model has seen the other's weights or training trajectory.

| Comparison | n | Spearman ρ | p |
|------------|---|-----------|---|
| Python concept fraction | 58 | 0.638 | 7.0 × 10⁻⁸ |
| Rust concept fraction | 9 | 0.717 | 0.030 |
| Python circuit size | 58 | 0.722 | 1.7 × 10⁻¹⁰ |

Both models agree that ImportFrom, Import, and Assert are the most concept-specific Python concepts. The Modular sub-group alone shows ρ = 0.89. Both models agree that builtins have near-zero signal. The ranking — which concepts earn dedicated circuitry and how much — is a property of the programming language, not the training run.

**[Figure 1: Concept fraction scatter, Qwen vs DeepSeek (Python, n = 58). Modular cluster in upper-right, builtins along the origin.]**

### 4.3 The "Where" Diverges Across Models

The depth at which concept-specific processing occurs is completely uncorrelated between models. Qwen concentrates concept-only signal in layers 14–19 (peak concept fraction 0.072 at L19). DeepSeek peaks in layers 4–8 (peak 0.267 at L7 — 5.5× higher than Qwen at the same layers). Layer-profile Spearman: ρ = −0.14 (not significant).

Within each model, the depth schedule is internally consistent: Qwen processes both Python and Rust at the same layers (circuit-size cross-language ρ = 0.97). The model has one depth strategy regardless of language.

**[Figure 2: Layer-resolved concept fraction profiles for Python — Qwen (peaks L14–19) vs DeepSeek (peaks L4–8). Two-panel figure, shared y-axis.]**

### 4.4 Interpretation

The mechanistic interpretability community has been asking "are circuits universal?" as a binary question. This data shows the question is ill-posed. Circuit structure has at least two independent axes:

- **What** earns dedicated circuitry is task-determined. It transfers across models.
- **Where** concepts are processed is training-determined. It does not transfer.

Concept identity and processing depth are orthogonal. A finding about one dimension says nothing about the other. Future studies of circuit universality should specify which dimension is under investigation.

## 5. Finding 2: Language Design Determines Representation Depth (~1.5 pages)

### 5.1 Overall Language Contrast

| | Modular | Non-modular | Object/Builtin | Overall |
|---|---------|-------------|----------------|---------|
| Python (Qwen) | 0.314 | 0.090 | 0.024 (Builtin) | 0.074 |
| Rust (Qwen) | 0.294 | 0.222 | 0.199 (Object) | 0.215 |
| Python (DeepSeek) | 0.164 | 0.172 | 0.068 (Builtin) | — |

Rust's overall mean concept fraction (0.215) is nearly 3× Python's (0.074) within the same model (Qwen). The difference is driven by two factors:

**(a) Python's sharp hierarchy collapses in Rust.** In Python, the Modular group (0.314) dominates; Non-modular (0.090) and Builtin (0.024) trail far behind — a 13:1 ratio. In Rust, all three groups are within 1.5:1 of each other. Rust's stricter syntax makes all concepts structurally distinctive, not just the atomic ones.

**(b) Rust's biggest boosts are for Python's weakest concepts.** Cross-language comparison for equivalent concepts within Qwen:

| Concept | Python CF | Rust CF | Boost |
|---------|-----------|---------|-------|
| Return | 0.043 | 0.305 | 7.2× |
| For | 0.063 | 0.319 | 5.1× |
| If | 0.070 | 0.305 | 4.4× |
| ClassDef / Struct | 0.035 | 0.131 | 3.7× |
| While | 0.277 | 0.329 | 1.2× |
| Break | 0.224 | 0.342 | 1.5× |
| Continue | 0.247 | 0.302 | 1.2× |
| Import / Use | 0.379 | 0.287 | 0.8× |

The pattern is clear: concepts already strong in Python (While, Break, Continue — all atomic, low-ambiguity) show modest boosts. Concepts weak in Python (Return, For, If — high token ambiguity, flexible syntax) show massive boosts. Import/Use is the one case where Python exceeds Rust — Python's `import` has uniquely distinctive syntax.

### 5.3 Cross-Language Neuron Sharing Confirms Structural Abstraction

The concept-fraction boost from Python to Rust raises the question: does the model build entirely separate circuitry per language, or does it reuse neurons for equivalent concepts? Cross-language sharing analysis (E7) shows that 6 of 7 equivalence classes share >10% of concept-only neurons between Python and Rust within Qwen:

| Equivalence class | Mean sharing | Peak layer | Peak |
|-------------------|-------------|------------|------|
| Iteration (For, While ↔ for) | 0.340 | 1 | 1.00 |
| Loop control (Break, Continue) | 0.289 | 1 | 1.00 |
| Branching (If ↔ if, match) | 0.271 | 7 | 1.00 |
| Return | 0.177 | 14 | 1.00 |
| Type def (ClassDef ↔ struct, enum) | 0.146 | 1 | 1.00 |
| Module import (Import ↔ use, mod) | 0.129 | 22 | 0.57 |
| Function def (FunctionDef ↔ fn) | 0.038 | 23 | 0.50 |

Iteration and loop control show the highest sharing (29–34%), consistent with these being the most syntactically parallel constructs across languages. Function def is the sole class below threshold (3.8%): Python `def` and Rust `fn` have very different syntactic structures (indentation- vs. brace-scoped, optional vs. mandatory return type), so the model correctly treats them as different computations.

The sharing pattern is interpretable: constructs that look and behave similarly across languages share neurons; constructs that merely serve the same abstract role but differ syntactically do not. This is structural sharing, not semantic aliasing.

### 5.2 Top Concepts

**Python (Qwen):** ImportFrom (0.478), Import (0.379), Assert (0.349), While (0.277), Continue (0.247), Raise (0.230), Try (0.225), Break (0.224), Pass (0.208). Below 0.10: all builtins except `open`.

**Rust (Qwen):** Crate (0.418), Super (0.409), Match (0.360), Break (0.342), Self (0.337), Loop (0.329), While (0.329), For (0.319), Return (0.305), If (0.305), Continue (0.302), Await (0.297), Use (0.287), HashMap (0.275).

Rust's top concepts include module-system keywords (Crate, Super, Use), control-flow (Match, Loop, While, For, If, Break, Continue), and self-reference (Self). No Rust concept falls below 0.10 — even the weakest Rust concepts have stronger concept-only signal than most Python concepts.

### 5.4 The DeepSeek Contrast

DeepSeek shows a flatter tier structure for Python: Modular (0.164), Non-modular (0.172), Builtin (0.068). The sharp Modular dominance in Qwen (0.314 vs 0.090) becomes nearly equal in DeepSeek. DeepSeek's top Python concepts include With (0.331), ImportFrom (0.324), ClassDef (0.280), Try (0.249) — a substantially different ordering from Qwen, though both models agree the atomicity cluster is near the top.

## 6. Finding 3: Architecture-Invariant Organisation and Semantic Clustering (~1 page)

### 6.1 The Atomicity Super-Cluster Replicates

The flow typology classifies each concept by its circuit-size-across-layers profile:

| | Python (Qwen) | Rust (Qwen) | Python (DeepSeek) |
|---|--------------|-------------|-------------------|
| two_phase | 7 | 2 | 33 |
| late_emergence | 95 | 71 | 4 |
| unclassified | 4 | 2 | 21 |

In Qwen, the Python two_phase concepts are: Assert, Break, Continue, Import, ImportFrom, Pass, Starred — exactly the atomicity super-cluster from CSP-Atlas plus Starred. The Rust two_phase concepts are Super and Use — both module-system keywords, the functional analogue of Python's Import/ImportFrom. This replication across three architectures (sparse 8-layer, dense 28-layer Qwen, and partially in DeepSeek) is strong evidence that the grouping is task-determined.

**DeepSeek disrupts the flow classification.** 33 Python concepts are two_phase in DeepSeek vs. 7 in Qwen. Flow type agreement between models: 57% (33/58). The "where" dimension (depth of processing) is model-specific, and flow type depends on "where," not "what." This is consistent with the What/Where dissociation.

### 6.2 Rust Semantic Clustering

With 75 concepts, hierarchical clustering at layer 14 reveals four semantically coherent groups:

1. **Type-system traits** (orange): Enum, Send, Option, Iterator, Copy, Eq, Drop, Debug, ToString.
2. **Memory/ownership + containers** (green): Pin, Mutex, Arc, Box, Vec, Impl, Async, Move, Future.
3. **Data definition** (red): Struct, Let, String, Trait, Default, Display, Hash, Return, Read.
4. **Control-flow + module** (purple): Fn, Pub, Use, Crate, While, Break, Loop, If, Match.

These correspond to genuine Rust semantic categories. The model has discovered the language's own conceptual taxonomy without being told it exists. This is the first demonstration of emergent semantic clustering for Rust concepts in a language model's internal representations.

### 6.3 Python Meta-Circuits (Qwen, L14)

Two main clusters emerge. The first contains builtins with near-zero concept fractions — a "poverty cluster" where high Jaccard values reflect tiny concept-only sets rather than genuine shared computation. The second contains AST nodes with real signal: earliest merges are (Assert, Pass) and (Continue, Break), confirming the atomicity grouping. Control-flow sub-clusters: (Try, While, With) and (For, AsyncFor).

## 7. Causal Validation (~0.5 page)

The decomposition is not epiphenomenal. Per-layer zero-ablation of concept-only neurons on Qwen yields 31 Bonferroni-significant concept × layer combinations across 6 of 7 testable AST concepts (Assert, Try, For, While, If, FunctionDef). Effects cluster in layers 13–19, coinciding with the correlational concept-fraction peaks.

Concept-only and shared neurons contribute roughly equally to keyword prediction (16:15 split in dominance across significant pairs). This is expected: gradient descent has no incentive to enforce clean separation beyond what disambiguation requires. The existence of *any* dedicated concept machinery — neurons with no token-level justification for firing — is evidence of genuine structural representation emerging without explicit supervision.

Negative controls confirm the decomposition: builtins (len, range) and Lambda have zero concept-only neurons at ε = 0.5, correctly identifying token-driven concepts.

## 8. Discussion (~0.75 page)

**Methodology contribution.** The pipeline is language-agnostic: adding a new language requires only a concept space definition and contrastive prompts. The decomposition produces directly comparable measurements across any (language, model) pair. We release all neuron lists, result CSVs, and analysis code.

**The What/Where dissociation.** This reframes the universality debate in mechanistic interpretability. Asking "are circuits universal?" conflates two dimensions. Concept identity transfers; processing depth does not. Future work should specify which dimension is under investigation.

**Language design as a predictor.** The concept-only fraction is predictable from two properties of the language: (a) how ambiguous the concept's token is (does the keyword appear in non-structural contexts?), and (b) how structurally distinctive the concept is (does it require unique syntactic patterns?). When both are high, concept fraction is high. When token ambiguity is zero (ownership operators `&`, `*`, `'`), concept fraction drops to near-zero — as predicted. This principle is testable for any new language.

**DeepSeek's earlier processing.** DeepSeek concentrates concept-specific work in layers 4–8 with 5.5× higher concept fractions than Qwen in those layers. Possible explanations: different training data composition (2T vs 5.5T tokens), different optimisation trajectories, or architectural differences in MLP activation function. Distinguishing these requires further controlled experiments.

**SwiGLU and the consistency threshold collapse.** The finding that consistency filtering is a no-op for SwiGLU MLPs has methodological implications: the 3 × 3 parameter grid used in CSP-Atlas reduces to a 3 × 1 sweep for modern architectures. Future work should replace the consistency dimension with a finer epsilon grid.

**Limitations.** Neuron-level analysis only — misses distributed representations that span many low-amplitude neurons. Two languages (Rust × Python), though linguistically diverse. Rust on DeepSeek not yet run (3 of 4 cells in the 2 × 2 matrix complete). Late-layer saturation at ε = 0.5 collapses concept fractions beyond layer 22. Causal validation on Qwen only; DeepSeek ablation pending.

**Future work.** Stage 2 adds a vector-level view (continuous activation directions, linear probes) to detect distributed representations invisible to the neuron view, a third language (Lean — enabling Curry-Howard correspondence analysis), and superposition measurement.
